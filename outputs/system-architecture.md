# เอกสารโครงสร้างระบบฝึกงาน / CWIE ภาควิชาไอที

## 1. ภาพรวมระบบ

ระบบนี้เป็นแพลตฟอร์มกลางสำหรับจัดการกระบวนการฝึกงานและ CWIE ตั้งแต่การลงทะเบียนนักศึกษาและสถานประกอบการ การจับคู่นักศึกษากับตำแหน่งฝึกงาน การติดตามเอกสาร CWIE การบันทึก logbook การประเมินผล และการตรวจสอบกิจกรรมย้อนหลัง

เป้าหมายหลักของระบบคือทำให้ทุกบทบาทเห็นงานที่ตนรับผิดชอบชัดเจน ลดงานติดตามเอกสารด้วยมือ และเก็บข้อมูลเชิงคุณภาพของสถานประกอบการเพื่อใช้ตัดสินใจในรอบถัดไป

## 2. บทบาทผู้ใช้งาน

| บทบาท | หน้าที่หลัก |
|---|---|
| เจ้าหน้าที่ / ผู้ประสานงาน | จัดการข้อมูลนักศึกษาและสถานประกอบการ, ติดตามการจับคู่, ติดตามเอกสาร CWIE, กำหนดช่วงเวลาประเมิน, ดูสถานะพี่เลี้ยง, ดูรายงานและ audit log |
| นักศึกษา | เพิ่ม/แก้ไขข้อมูลส่วนตัวและข้อมูลติดต่อของตนเอง, ดูข้อมูลการฝึกงาน, บันทึก daily logbook, ส่งสรุปรายสัปดาห์, ส่งแบบประเมินสถานประกอบการเมื่ออยู่ในช่วงที่เปิด, ติดตามสถานะเอกสารของตน |
| อาจารย์นิเทศ | ดูรายการนิเทศที่ได้รับมอบหมาย, เพิ่ม/แก้ไขข้อมูลติดต่อและความเชี่ยวชาญของตนเอง, ประเมินสถานประกอบการเมื่ออยู่ในช่วงที่เปิด, บันทึกผลนิเทศ |
| สถานประกอบการ | เปิดรับตำแหน่งฝึกงาน, ยืนยันเอกสาร CWIE, จัดการพี่เลี้ยง, ผูกนักศึกษากับพี่เลี้ยง, ตรวจ logbook, ประเมินนักศึกษา |

หมายเหตุสำคัญ: การเพิ่ม/แก้ไข/ผูกพี่เลี้ยงเป็นความรับผิดชอบของสถานประกอบการ เจ้าหน้าที่มีหน้าที่ติดตามและส่งคำขอให้บริษัทอัปเดตข้อมูลเท่านั้น

## 3. โมดูลหลักของระบบ

### 3.1 Authentication / Role Portal

ระบบแยกหน้าการใช้งานตามบทบาทหลังเข้าสู่ระบบ

- เจ้าหน้าที่เข้าสู่ staff dashboard
- นักศึกษาเข้าสู่ student portal
- อาจารย์นิเทศเข้าสู่ advisor portal
- สถานประกอบการเข้าสู่ company portal

ในระบบจริงควรรองรับ SSO ของมหาวิทยาลัยสำหรับเจ้าหน้าที่ อาจารย์ และนักศึกษา ส่วนสถานประกอบการอาจใช้บัญชีอีเมลบริษัทพร้อมการยืนยันตัวตนแยกต่างหาก

Student, Advisor และ Company portal ควรใช้ shell เดียวกับแดชบอร์ดเจ้าหน้าที่ คือ sidebar สีเข้มด้านซ้าย, topbar, พื้นหลังกลางสีอ่อน และ card/content pattern ชุดเดียวกัน เพื่อให้ผู้ใช้ทุกบทบาทรู้สึกว่าอยู่ในระบบเดียวกัน แต่เมนูและสิทธิ์ยังแยกตามบทบาท เช่น ภาพรวม ข้อมูลส่วนตัว/ข้อมูลบริษัท Logbook/งานนิเทศ เอกสาร ประเมิน และร้องเรียน/แจ้งปัญหา

นักศึกษา อาจารย์นิเทศ และสถานประกอบการต้องผ่านการอนุมัติบัญชีโดยเจ้าหน้าที่ก่อนเข้าใช้งาน portal ของตนเองได้ ระบบต้องมีคิวคำขอเข้าใช้งาน แสดงหลักฐาน/ข้อมูลยืนยันตัวตน สถานะรออนุมัติ/อนุมัติแล้ว/ปฏิเสธ และบันทึก audit log ทุกครั้งที่เจ้าหน้าที่อนุมัติหรือปฏิเสธบัญชี

Advisor portal ต้องมีหน้า self-service profile ให้อาจารย์นิเทศเพิ่ม/แก้ไขข้อมูลของตนเอง เช่น อีเมล เบอร์โทร ห้องพัก เวลาพบ ช่องทางติดต่อสำรอง ความเชี่ยวชาญ และจำนวนรับนิเทศสูงสุด เจ้าหน้าที่ใช้ข้อมูลนี้เพื่อดูแล workflow ได้ แต่ไม่ควรเป็นผู้แก้ไขข้อมูลส่วนตัวแทนอาจารย์ในงานปกติ

Student portal ต้องมีหน้า self-service profile ให้นักศึกษาเพิ่ม/แก้ไขข้อมูลของตนเอง เช่น เบอร์โทร อีเมล Line ที่อยู่ ผู้ติดต่อฉุกเฉิน ทักษะหลัก Portfolio และเป้าหมายสายงาน ข้อมูลนี้ใช้ใน Student 360, เอกสาร CWIE, การจับคู่ และการติดต่อระหว่างฝึกงาน

### 3.2 Dashboard เจ้าหน้าที่

ใช้สรุปภาพรวมรอบฝึกงาน เช่น

- จำนวนนักศึกษาทั้งหมด
- จำนวนที่จับคู่แล้ว / รอจับคู่
- จำนวนสถานประกอบการ
- บริษัทที่ติดธงเฝ้าระวัง
- งานที่ต้องดำเนินการเร่งด่วน

### 3.3 นักศึกษา

ใช้จัดการรายชื่อนักศึกษาที่ออกฝึกงานในรอบปัจจุบัน

ข้อมูลหลัก:

- รหัสนักศึกษา
- ชื่อ-นามสกุล
- ประเภทฝึกงาน เช่น สหกิจ / ฝึกงานทั่วไป
- ทักษะหลัก
- สถานะการจับคู่
- สถานประกอบการที่ได้รับมอบหมาย

### 3.3.1 นักศึกษา 360°

เป็นหน้ารวมข้อมูลรายคน ใช้เป็นจุดอ้างอิงกลางเมื่อเจ้าหน้าที่ อาจารย์ หรือผู้เกี่ยวข้องต้องติดตามนักศึกษาหนึ่งคน

ข้อมูลที่ควรรวมในหน้าเดียว:

- ข้อมูลนักศึกษาและทักษะ
- สถานประกอบการที่จับคู่
- พี่เลี้ยงบริษัท
- อาจารย์นิเทศ
- สถานะเอกสาร CWIE รายขั้น
- สถานะ daily logbook และ weekly summary
- ประวัติ audit/history ที่เกี่ยวข้องกับนักศึกษารายนั้น
- แถบ Action Required เพื่อระบุว่างานค้างอยู่ที่ใคร
- Risk / Early Warning ที่เปิดอยู่ของนักศึกษารายนั้น
- Comment thread กลางสำหรับการประสานงานรายคน

ตัวอย่าง Action Required:

- ค้างที่สถานประกอบการ: ยังไม่ส่ง Job plan
- ค้างที่พี่เลี้ยง: logbook รอตรวจ
- ค้างที่เจ้าหน้าที่: หนังสือส่งตัวรอลายเซ็น
- ค้างที่นักศึกษา: ยังไม่ส่งรายงานหรือยังไม่ประเมิน

### 3.3.2 Risk / Early Warning

ระบบควรมี early warning สำหรับเหตุการณ์ที่ต้องติดตามระหว่างฝึกงาน เช่น logbook ค้างเกินกำหนด ยังไม่ผูกพี่เลี้ยง ยังไม่ยืนยันนัดนิเทศ เอกสาร CWIE เลยกำหนด หรือบริษัทติดธงเฝ้าระวัง

ทุก risk ต้องมีระดับความเสี่ยง เจ้าของงาน กำหนดติดตาม สถานะเปิด/ปิด และ audit log เมื่อมีการปิดความเสี่ยง

### 3.3.3 Supervision Schedule

เจ้าหน้าที่และอาจารย์นิเทศต้องเห็นตารางนัดนิเทศรายนักศึกษา รองรับสถานะรอยืนยัน ยืนยันแล้ว ต้องเลื่อน และนิเทศแล้ว เพื่อให้การประสานงานระหว่างอาจารย์ บริษัท และนักศึกษาไม่หลุดจาก workflow

### 3.3.4 Change Request Workflow

ข้อมูลบางประเภทควรให้ผู้ใช้ส่งคำขอแก้ไขและให้เจ้าหน้าที่อนุมัติก่อนเปลี่ยนจริง เช่น ชื่อ-นามสกุล ประเภทการฝึก สถานประกอบการ อาจารย์นิเทศ เบอร์โทรสำคัญ และข้อมูลผู้ประสานงานบริษัท ทุกคำขอต้องเก็บค่าเดิม ค่าใหม่ เหตุผล หลักฐาน ผู้อนุมัติ และ audit log

### 3.3.5 Issue / Complaint Reports

นักศึกษาและสถานประกอบการควรแจ้งปัญหาระหว่างฝึกงานได้จาก portal ของตน เช่น งานไม่ตรงสาย ไม่มีพี่เลี้ยง ปัญหาความปลอดภัย ปัญหา logbook หรือการสื่อสารกับบริษัท ระบบต้องมีระดับความเร่งด่วน ความลับ เจ้าของเคส สถานะ และประวัติการปิดเคส

สิทธิ์การเห็นเรื่องร้องเรียน:

- เจ้าหน้าที่เห็นทุกเคสและเป็นผู้ triage / ปิดเคส
- นักศึกษาเห็นและติดตามเฉพาะเคสของตนเอง
- สถานประกอบการเห็นเฉพาะเคสที่เกี่ยวข้องกับนักศึกษาของบริษัท
- อาจารย์นิเทศเห็นเฉพาะเคสของนักศึกษาที่ตนเองรับผิดชอบ

### 3.3.6 Advisor Workload

เจ้าหน้าที่ควรเห็นภาระงานอาจารย์นิเทศรายคน เช่น จำนวนนักศึกษาที่ดูแล จำนวนรอประเมิน นัดนิเทศที่ต้องประสาน Risk ของนักศึกษาที่ดูแล และ capacity เพื่อช่วยกระจายงานนิเทศอย่างเหมาะสม

### 3.4 สถานประกอบการ

ใช้จัดการฐานข้อมูลบริษัทและเงื่อนไขรับนักศึกษา

ข้อมูลหลัก:

- ชื่อบริษัท
- รูปแบบการทำงาน เช่น Onsite / Hybrid / Remote
- ที่ตั้ง
- ประเภทที่รับ เช่น สหกิจ / ฝึกงานทั่วไป
- ความจุรอบปัจจุบัน
- Tech stack / เครื่องมือ
- คะแนนประเมินสะสม
- สถานะ เช่น แนะนำ / ใช้ได้ / เฝ้าระวัง / ใหม่
- ข้อมูลพี่เลี้ยง

บริษัทใหม่ต้องแสดงสถานะ "ยังไม่มีคะแนน" และไม่ควรถูกเติมคะแนนเริ่มต้นปลอม เช่น 4.0 โดยอัตโนมัติ

### 3.5 จับคู่ฝึกงาน

ระบบช่วยจัดอันดับสถานประกอบการที่เหมาะกับนักศึกษา โดยพิจารณาจาก

- ประเภทการฝึกงานที่บริษัทรับ
- ทักษะของนักศึกษาเทียบกับ tech stack ของบริษัท
- ความจุที่ยังว่าง
- คะแนนคุณภาพจากรอบก่อน
- สถานะเฝ้าระวัง
- เงื่อนไขพิเศษ เช่น NDA

กฎสำคัญ:

- ห้ามจับคู่เกินความจุบริษัท
- บริษัทที่เต็มต้องปิด action การมอบหมาย
- บริษัทเฝ้าระวังไม่ควรถูกมอบหมายจนกว่าจะตรวจสอบ/ปลดธง
- บริษัทใหม่ไม่มีคะแนนปลอม แต่ยังเสนอได้โดยแสดงว่า "ยังไม่มีคะแนน"
- การจับคู่ทักษะต้องรองรับ synonym เช่น `ML` = `Machine Learning`, `K8s` = `Kubernetes`, `Node` = `Node.js`

### 3.6 วงจรเอกสาร CWIE

โมดูลนี้เป็นหัวใจของระบบ ใช้ติดตามเอกสารรายคนและรู้ว่าเอกสารค้างอยู่ที่ใคร

ตัวอย่าง flow เอกสาร:

1. ใบสมัคร / โปรไฟล์นักศึกษา
2. บริษัทรับรองตำแหน่งงาน
3. หนังสือส่งตัว CWIE
4. Job plan / แผนฝึกงานจากพี่เลี้ยง
5. บันทึกนิเทศกลางภาค
6. รายงานปิดเล่ม / เอกสารจบกระบวนการ

แต่ละเอกสารควรมีข้อมูล:

- เจ้าของงาน เช่น นักศึกษา / บริษัท / เจ้าหน้าที่ / อาจารย์
- สถานะ เช่น รอดำเนินการ / รอลายเซ็น / ส่งแล้ว / เกินกำหนด
- วันครบกำหนด
- ไฟล์แนบ
- ประวัติการเปลี่ยนแปลง

### 3.7 Logbook นักศึกษา

นักศึกษาใช้บันทึกการปฏิบัติงานรายวัน และระบบรวมเป็นสรุปรายสัปดาห์เพื่อส่งให้พี่เลี้ยงตรวจ

ข้อมูล Daily Log หลัก:

- วันที่
- เวลาเริ่ม / เวลาสิ้นสุด / ชั่วโมงรวม
- รูปแบบงาน เช่น Onsite / Hybrid / Remote
- งานที่ปฏิบัติจริง
- ผลลัพธ์หรือหลักฐาน เช่น PR, commit, ticket, เอกสาร, test case
- ปัญหา / อุปสรรค
- สิ่งที่ได้เรียนรู้
- แผนงานวันถัดไป
- ไฟล์แนบหรือหลักฐาน
- สถานะการตรวจจากพี่เลี้ยง

สถานะ daily log:

- ยังไม่บันทึก
- แบบร่าง
- ส่งแล้ว
- ขอแก้ไข
- อนุมัติแล้ว
- ล็อกแล้ว

Weekly Summary:

- ระบบรวมข้อมูล daily log ในสัปดาห์นั้น
- นักศึกษากดส่งสรุปสัปดาห์ให้พี่เลี้ยงตรวจ
- พี่เลี้ยงอนุมัติหรือขอแก้ไขเป็นรายวันได้
- อาจารย์นิเทศและเจ้าหน้าที่ดู summary เพื่อประกอบการติดตาม

พี่เลี้ยงของสถานประกอบการเป็นผู้ตรวจและอนุมัติ logbook ส่วนอาจารย์และเจ้าหน้าที่สามารถดูเพื่อประกอบการติดตามได้

### 3.8 ระบบพี่เลี้ยง

แบ่งหน้าที่เป็น 2 ส่วน:

ฝั่งสถานประกอบการ:

- เพิ่มพี่เลี้ยง
- แก้ไขชื่อ ตำแหน่ง อีเมล เบอร์โทร
- ผูกนักศึกษากับพี่เลี้ยงที่รับผิดชอบ
- ตรวจ logbook ของนักศึกษาที่ดูแล
- ประเมินนักศึกษา

ฝั่งเจ้าหน้าที่:

- ดูว่านักศึกษามีพี่เลี้ยงหรือยัง
- ดูสถานะ logbook ที่รอตรวจ
- ส่งคำขอให้สถานประกอบการอัปเดตข้อมูล
- ไม่แก้ไขข้อมูลพี่เลี้ยงแทนบริษัท

### 3.9 พอร์ทัลสถานประกอบการ

เป็นพื้นที่ทำงานของบริษัทโดยตรง

ฟีเจอร์หลัก:

- ดูตำแหน่งที่เปิดรับ
- ดูคิวงานของสถานประกอบการ เช่น ผูกพี่เลี้ยง ตรวจ logbook ส่ง Job plan ประเมินนักศึกษา
- ดูนักศึกษาที่มาฝึกงาน
- จัดการพี่เลี้ยง
- ผูกนักศึกษากับพี่เลี้ยง
- ส่ง Job plan
- ตรวจ logbook
- ประเมินนักศึกษา
- ยืนยันเอกสาร CWIE ที่เกี่ยวข้อง

### 3.9.1 Company Task Queue

หน้าแรกของสถานประกอบการควรมีคิวงานที่ต้องดำเนินการ เพื่อให้บริษัทเห็น action สำคัญทันทีหลังเข้าสู่ระบบ

ตัวอย่างคิวงาน:

- ผูกพี่เลี้ยงให้นักศึกษา
- ตรวจ logbook ที่รออนุมัติ
- ส่ง Job plan ภายในกำหนด
- ยืนยัน mentor ผู้ประเมินปลายภาค
- ประเมินนักศึกษาเมื่อถึงช่วงเวลา

### 3.9.2 Company Workflow Tabs

พอร์ทัลสถานประกอบการควรแบ่งงานเป็น tab หลัก:

- ตำแหน่งที่เปิดรับ
- พี่เลี้ยง
- นักศึกษาที่ดูแล
- เอกสาร CWIE
- Logbook
- ประเมิน

### 3.10 ระบบแจ้งเตือน

ใช้แจ้งเตือนงานที่ต้องดำเนินการ เช่น

- เอกสาร CWIE ใกล้ครบกำหนด
- หนังสือส่งตัวรอลายเซ็น
- Job plan ยังไม่ส่ง
- บริษัทเต็ม ไม่สามารถจับคู่เพิ่ม
- logbook รอตรวจ
- ผลประเมินเข้าใหม่

ควรมี notification ใน topbar สำหรับเจ้าหน้าที่ และ notification ตามบทบาทใน portal อื่น ๆ

### 3.11 ตั้งค่าช่วงเวลาประเมิน

เจ้าหน้าที่เป็นผู้กำหนดช่วงเวลาเปิด/ปิดแบบประเมินของแต่ละบทบาท ระบบต้องใช้ช่วงเวลานี้ควบคุมการแสดงฟอร์มประเมินใน portal ของนักศึกษาและอาจารย์นิเทศ

ประเภทช่วงเวลาประเมิน:

- นักศึกษาประเมินสถานประกอบการ
- อาจารย์นิเทศประเมินสถานประกอบการ

กฎการทำงาน:

- หากยังไม่ถึงช่วงเวลา ให้แสดงการ์ดล็อกพร้อมวันที่เปิด/ปิด
- หากอยู่ในช่วงเวลา ให้แสดงฟอร์มประเมินและอนุญาตให้ส่ง
- หากเลยวันปิด ให้แสดงสถานะปิดแล้วและไม่อนุญาตให้ส่ง
- การเปลี่ยนช่วงเวลาต้องถูกบันทึกใน audit log
- ควรแจ้งเตือนผู้เกี่ยวข้องก่อนปิดช่วงประเมิน เช่น 7 วัน / 3 วัน / 1 วัน

### 3.12 Audit Log

บันทึกกิจกรรมสำคัญเพื่อย้อนตรวจสอบ

ตัวอย่างกิจกรรม:

- เข้าสู่ระบบ
- เพิ่ม/แก้ไขข้อมูลนักศึกษา
- เพิ่ม/แก้ไขสถานประกอบการ
- จับคู่หรือยกเลิกการจับคู่
- ระบบป้องกันการจับคู่เกินความจุ
- บริษัทเพิ่ม/แก้ไขพี่เลี้ยง
- บริษัทผูกนักศึกษากับพี่เลี้ยง
- ส่ง/อนุมัติเอกสาร CWIE
- ส่ง/อนุมัติ logbook
- เจ้าหน้าที่กำหนดหรือแก้ไขช่วงเวลาประเมิน
- ส่งผลประเมิน

ข้อมูล audit log ควรมี:

- เวลา
- ผู้ใช้งาน
- บทบาท
- ประเภทกิจกรรม
- รายละเอียด
- IP / device
- before/after value สำหรับข้อมูลสำคัญ

## 6. Data Flow สำคัญ

### 6.1 การจับคู่

1. เจ้าหน้าที่เลือกนักศึกษา
2. ระบบคำนวณบริษัทที่รับประเภทฝึกงานตรงกัน
3. ระบบเทียบทักษะด้วย synonym
4. ระบบตัดบริษัทที่เต็มหรือเฝ้าระวังตามกฎ
5. เจ้าหน้าที่มอบหมายบริษัท
6. ระบบอัปเดต capacity
7. ระบบสร้างงานต่อเนื่อง เช่น เอกสาร CWIE และการรอพี่เลี้ยงจากบริษัท

### 6.2 การจัดการพี่เลี้ยง

1. บริษัทเข้าสู่ company portal
2. บริษัทเพิ่ม/แก้ไขข้อมูลพี่เลี้ยง
3. บริษัทผูกนักศึกษากับพี่เลี้ยง
4. ระบบบันทึก mentor assignment
5. นักศึกษาเห็นพี่เลี้ยงใน portal
6. พี่เลี้ยงตรวจ logbook และประเมินนักศึกษา
7. เจ้าหน้าที่เห็นสถานะเพื่อใช้ติดตาม

### 6.3 การส่ง Logbook

1. นักศึกษาบันทึก daily log ในแต่ละวัน
2. นักศึกษาบันทึกแบบร่างหรือส่งรายวันได้
3. เมื่อครบสัปดาห์ ระบบรวม daily log เป็น weekly summary
4. นักศึกษากดส่งสรุปรายสัปดาห์ให้พี่เลี้ยงตรวจ
5. ระบบแจ้งเตือนไปยังพี่เลี้ยงที่รับผิดชอบ
6. พี่เลี้ยงตรวจ อนุมัติ หรือขอแก้ไขเป็นรายวัน
7. ระบบบันทึกสถานะและ audit log
8. อาจารย์นิเทศและเจ้าหน้าที่ดูประกอบการติดตามได้

### 6.4 วงจรเอกสาร CWIE

1. นักศึกษาส่งโปรไฟล์
2. บริษัทรับรองตำแหน่งและพี่เลี้ยง
3. เจ้าหน้าที่ออกหนังสือส่งตัว
4. บริษัทส่ง Job plan
5. อาจารย์นิเทศติดตามกลางภาค
6. นักศึกษาส่งรายงานปิดเล่ม
7. ระบบปิดวงจรเอกสารและเก็บหลักฐาน

### 6.5 การตั้งค่าช่วงเวลาประเมิน

1. เจ้าหน้าที่เปิดหน้า "ตั้งค่าช่วงเวลาประเมิน"
2. เจ้าหน้าที่กำหนดวันที่เปิด/ปิดสำหรับนักศึกษาประเมินสถานประกอบการ
3. เจ้าหน้าที่กำหนดวันที่เปิด/ปิดสำหรับอาจารย์นิเทศประเมินสถานประกอบการ
4. ระบบบันทึก evaluation window และ audit log
5. เมื่อผู้ใช้เข้าสู่ portal ระบบตรวจวันที่ปัจจุบันกับช่วงเวลาที่กำหนด
6. ถ้าอยู่ในช่วงเปิด ระบบแสดงฟอร์มประเมิน
7. ถ้ายังไม่ถึงช่วงหรือเลยวันปิด ระบบแสดงการ์ดล็อกและไม่อนุญาตให้ส่งประเมิน
8. ระบบแจ้งเตือนผู้เกี่ยวข้องก่อนช่วงประเมินปิด

## 7. ข้อกำหนดด้านความถูกต้องของระบบ

- ต้องไม่จับคู่เกิน capacity
- ต้องไม่ใช้คะแนนปลอมกับบริษัทใหม่
- ต้องแยก ownership ของข้อมูลพี่เลี้ยงให้ชัดเจน
- ต้องเก็บ audit log ของ action สำคัญ
- ต้องแสดงสถานะเอกสารรายคน ไม่ใช่แค่ภาพรวม
- ต้องมี notification สำหรับงานที่ใกล้ครบกำหนดหรือค้าง
- ต้องรองรับการค้นหาและกรองตามรอบฝึกงาน
- แบบประเมินของนักศึกษาและอาจารย์นิเทศต้องเปิดตามช่วงเวลาที่เจ้าหน้าที่กำหนดเท่านั้น
- Daily logbook ต้องไม่ถูกส่งย้อนหลังโดยไม่มี audit trail
- พี่เลี้ยงต้องเห็นรายการ daily log ที่รอตรวจและสามารถขอแก้ไขเป็นรายวันได้

## 8. แนวทางพัฒนาต่อ

ลำดับที่ควรพัฒนาก่อน:

1. Role-based authentication และ permission
2. ฐานข้อมูลนักศึกษา / สถานประกอบการ / พี่เลี้ยง
3. Matching engine พร้อม capacity guard และ synonym skill matching
4. CWIE document workflow รายคน
5. Company portal สำหรับจัดการพี่เลี้ยงและ Job plan
6. Student daily logbook และ weekly summary
7. Evaluation window และ gating สำหรับ student/advisor portal
8. Notification service
9. Evaluation และ company quality report
10. Audit log และ export report
